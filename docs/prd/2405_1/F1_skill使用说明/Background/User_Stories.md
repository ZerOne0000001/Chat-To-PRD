# 用户故事 (User Stories)

## 史诗 (Epic) 1: 提供全局视角与基础介绍
*作为一名需要了解 `pm-workflow` 技能的用户或演示者，我希望能在演示页面中清晰看到技能的基础介绍，以便快速建立对它的全局认知。*

**User Story 1.1: 技能简介模块**
*   **作为**：一名刚开始观看演示视频的用户
*   **我希望**：能在大纲页面最顶端看到 `pm-workflow` 的名称、一段简明扼要的功能简介（比如：它是一个“产品工作流主控节点”）。
*   **以便于**：我能立刻知道这个技能是用来干什么的。
*   **验收标准 (Acceptance Criteria)**:
    1.  页面顶端有显著的标题 `pm-workflow`。
    2.  包含一段描述其全流程调度与核心作用的简介文本。

**User Story 1.2: 触发条件说明**
*   **作为**：一名想要亲自尝试使用该技能的用户
*   **我希望**：能清楚地知道在什么情况下、输入什么指令才能唤起这个技能。
*   **以便于**：我在自己操作时能准确触发它。
*   **验收标准 (Acceptance Criteria)**:
    1.  简介下方有明确的“触发条件”模块。
    2.  列举常见的唤起方式（例如：“输入一句话需求”或“提供已有原型文件”）。

## 史诗 (Epic) 2: 场景分支与流程步骤展示
*作为一名演示者，我希望能在大纲页面结构化地展示不同场景下的具体流程，以便在录视频时作为目录随时切换和指引。*

**User Story 2.1: 场景分支的切换/展示**
*   **作为**：一名正在录制视频的演示者
*   **我希望**：页面能清晰区分“标准工作流”和“快速通道”两种不同的使用场景。
*   **以便于**：我可以在讲解时，向观众说明不同输入会导致 AI 走不同的流程。
*   **验收标准 (Acceptance Criteria)**:
    1.  页面包含“使用场景”模块。
    2.  明确划分出不同场景（例如：标准工作流、快速通道）及其对应的差异化流程骨架。

**User Story 2.2: 阶段步骤的序列化呈现**
*   **作为**：一名观看视频的小白用户
*   **我希望**：能看到以 `pm-workflow` 为核心的从 0 到 6 的各个阶段大纲（如：初始化 -> 需求启动 -> 架构设计 -> 原型制作 -> 文档生成 -> 收尾）。
*   **以便于**：我能清晰了解整个产品设计的生命周期。
*   **验收标准 (Acceptance Criteria)**:
    1.  以时间轴、卡片流或步骤条的形式，顺序展示各个阶段。
    2.  每个阶段都应包含阶段名称（如“阶段 1: pm-kickoff”）。

## 史诗 (Epic) 3: 产出物可视化与关联
*作为一名观众，我希望能清楚地知道 AI 在每个阶段到底生成了哪些具体的中间文件，以便理解整个体系的运转逻辑。*

**User Story 3.1: 阶段关联的产物介绍**
*   **作为**：一名正在学习 `pm-workflow` 的小白用户
*   **我希望**：在每个阶段的下方，能看到该阶段会生成哪些具体的中间文件名称（如 `System_Survey.md`）及其简单的介绍。
*   **以便于**：我能理解这些 Markdown 文件在整个工作流中扮演的角色。
*   **验收标准 (Acceptance Criteria)**:
    1.  在流程大纲的每一个对应阶段卡片或节点内，列出该阶段的“产出物”。
    2.  产出物仅需展示“文档名称”以及一句简短的“介绍”（不需要复杂的缩略图或代码预览）。
