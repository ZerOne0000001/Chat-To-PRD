# PM Prototype 技能 (原型设计)

**必须**使用此技能根据页面结构生成高保真的 HTML 原型。

## 执行工作流 (Execution Workflow)

### 1. 准备阶段 (Preparation)

- **必须读取**当前需求目录下的以下文件：
  - `Background/Requirement_Background.md`
  - `Background/User_Stories.md`
  - `Design/Page_Structure.md`
- **强制分析**：如果信息不足以支撑界面生成，你**必须**询问用户补充。信息充足后，描述关键页面拟定的 UI 布局和交互流程。

### 2. 视觉风格定义与样例确认 (核心前置阶段)

为确保生成过程符合预期，**你必须严格按照以下步骤顺序与用户交互，并在每一步等待用户回答后才能进入下一步，绝对禁止跳步或合并步骤！**

#### 步骤 2.1：分端推荐风格与尺寸确认
- **分析多端情况**：根据背景信息和页面结构，明确当前项目涉及哪些端（如 PC、H5、小程序、收银机等）。
- **强制设计思考**：对于每一个端，从以下维度进行审视：Purpose（解决什么问题）、Tone（极端调性）、Constraints（物理限制）、Differentiation（视觉抓手）。
- **询问动作**：
  1. 向用户**按端**分别推荐 5-10 个风格（绝对禁止平庸风格），要求用户为每个端**分别**选择一种风格（**必须**允许用户多选或自由组合）。
  2. 同时，**必须**主动询问用户每个端的“页面基础尺寸（如 1920x1080、375x812 等）”要求。
- **🛑 强制暂停**：输出推荐后，**必须在此处停止**，等待用户回复其选择的风格和尺寸。

#### 步骤 2.2：挑选样例页面
- **确认输入**：在收到用户对风格和尺寸的选择后，总结其决定。
- **询问动作**：向用户列出当前设计的几个核心页面，并**必须**询问：“您希望拿哪一个（或几个）页面作为初始样例去生成并验证风格效果？”
- **🛑 强制暂停**：输出问题后，**必须在此处停止**，等待用户指定样例页面。

#### 步骤 2.3：生成样例与迭代确认
- **生成样例**：使用 `frontend-design` 技能，根据用户指定的样例页面、选定的风格和尺寸，生成一份 HTML 样例文件（放在 `Prototypes/` 目录下）。
- **询问动作**：向用户展示生成的链接，并**必须**询问：“样例页面已生成。请在浏览器中查看效果，当前的视觉风格和布局是否符合您的预期？”
- **🛑 强制暂停**：输出问题后，**必须在此处停止**，等待用户确认或提出修改意见。
- **强制循环**：如果用户不满意，**必须**采访其感受，推荐新风格并重新生成样例，直到用户明确表示满意。

#### 步骤 2.4：生成设计倾向指南 (Design Trend Guide)
- 当用户对样例页面明确表示满意后，你**必须**在当前需求目录下的 `Design/` 文件夹中生成 `Design_Trend_Guide.md`。
- **🚨 核心约束**：你**必须**读取并严格按照 `[references/Example_Design_Trend.md](references/Example_Design_Trend.md)` 的骨架和颗粒度要求来编写这份指南。它是一份写给后续 `frontend-design` 的严苛作战指令，**绝对禁止**泛泛而谈。

### 3. 生成阶段 (Generation)

- **必须**在当前需求根目录下创建 `Prototypes/` 文件夹。
- **调用生成**：调用 `frontend-design` 技能，并严格依据前一阶段生成的 `Design_Trend_Guide.md` 生成高质量的 HTML/CSS/JS。
- **强制原型规范**：
  1. **必须**产出单文件 HTML 原型（包含内联 CSS/JS）。
  2. **必须**使用 Tailwind CSS（CDN 引入）。
  3. **必须**包含关键交互状态（通过原生 JS 或 URL Hash 实现：页面切换、同页状态切换、弹窗/抽屉等覆盖层）。
  4. 视觉表现**必须**死磕“设计倾向指南”，结构细节**必须**与 `Page_Structure.md` 100% 对齐。

### 4. 原型审查与迭代 (核心迭代循环)

- **强制汇报**：确保所有 HTML 文件在 `Prototypes/` 中生成完毕后，展示它们的本地链接（例如：`[home.html](Prototypes/home.html)`）。
- **强制询问**：引导用户在浏览器中打开验证，并询问：“原型已生成。请查看界面与交互，是否有需要调整的功能、逻辑或页面流转？”
- **🛑 强制暂停**：你**必须**在此处停止输出！等待用户反馈修改意见。
- **迭代修改**：根据用户反馈，仅修改 HTML 原型源码，直至用户表示“完全满意”或“确认无误”。

### 5. 逆向同步与严格一致性校对 (Reverse Sync & Consistency Audit)

当用户确认原型定稿后，你**必须**执行以下严格的逆向审计动作：

- [ ] **动作 1 (重新读取)**：**必须**使用文件读取工具，重新读取 `Prototypes/` 目录下最新定稿的 HTML 源码。
- [ ] **动作 2 (提取与比对)**：提取 HTML 源码中所有实际存在的独立页面、弹窗 (Modal)、抽屉 (Drawer)、交互卡片等核心 DOM 结构，与原有的 `Page_Structure.md` 进行逐一核对。
- [ ] **动作 3 (逆向修改)**：如果原型中**新增**了弹窗、**删除**了按钮，或**修改**了组件交互形式，你**必须**直接修改并覆写 `Page_Structure.md`，确保底层设计文档与最终代码结构达到 100% 镜像一致。
- [ ] **动作 4 (输出报告)**：同步完成后，你**必须**向用户输出简短的一致性报告（例如：“我已完成逆向同步，在 `Page_Structure.md` 中补充了 X，移除了 Y，当前文档与原型已 100% 对齐”）。
- **询问确认**：“原型已定稿，且底层设计文档已根据最终原型同步更新完毕。我们可以进入下一阶段（PRD 编写）了吗？”
- **🛑 强制停止**：在用户明确确认之前，你**绝对不要**继续执行任何其他工作流步骤。
